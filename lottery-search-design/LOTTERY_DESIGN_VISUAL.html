โจทย์ & คำตอบRequirement & Answer

ระบบค้นหา + จอง ticket จากที่สร้างไว้ล่วงหน้า 1 ล้านใบ user ค้นด้วย pattern 6 หลักที่มี wildcard แล้วได้ ticket ที่ตรงและยังไม่มีใครจองไป 1 ใบ โจทย์จริงๆ ไม่ใช่แค่ "หาให้เจอ" แต่คือการันตีว่าสองคนที่ค้น pattern เดียวกันพร้อมกัน จะไม่มีวันได้ ticket ใบเดียวกันเด็ดขาด A search-and-reserve system over 1,000,000 pre-generated tickets. A user searches with a 6-digit wildcard pattern and gets back one matching, unallocated ticket. The real challenge isn't just "finding a match" - it's guaranteeing that two users searching the same pattern at the same instant can never receive the same ticket.

Database

เลือก DB อะไร — PostgreSQL / Mongo / Redis / Elastic / ClickHouse แล้วอธิบายเหตุผล Which database? - PostgreSQL / Mongo / Redis / Elastic / ClickHouse, and explain why.

คำตอบAnswer PostgreSQL — SELECT ... FOR UPDATE SKIP LOCKED แก้ concurrency ได้ตรงจุดโดยไม่ต้องมี distributed lock แยก, generated column + B-tree รองรับ wildcard ได้เร็ว PostgreSQL - SELECT ... FOR UPDATE SKIP LOCKED solves the concurrency problem directly without a separate distributed lock; generated columns + B-tree indexes make wildcard matching fast.

Algorithm

Wildcard search หาอย่างไร — Prefix Index, Trie, Bitmap, Inverted Index, B-tree, Hash (ไม่ใช่แค่ LIKE '%23' เพราะข้อมูล 1 ล้านแถว) How do you search wildcards? - Prefix Index, Trie, Bitmap, Inverted Index, B-tree, Hash (not just LIKE '%23', given 1M rows).

คำตอบAnswer แยกเก็บตัวเลขแต่ละหลักของ ticket เป็นคอลัมน์ของตัวเอง (d0..d5) แล้วสร้าง index ให้ทุกคอลัมน์ เวลาค้นด้วย pattern ที่มีบางหลักระบุตัวเลขจริง Postgres จะเลือกใช้ index ของหลักนั้นให้เองโดยอัตโนมัติ ทำให้หาเร็วโดยไม่ต้องสร้างโครงสร้างข้อมูลพิเศษเพิ่มเติม Store each digit of the ticket in its own column (d0..d5) and put an index on every column. When a pattern fixes some digit, Postgres automatically picks that digit's index, making the search fast without building any extra data structure.

Concurrency

กัน race condition อย่างไร — SELECT ... FOR UPDATE SKIP LOCKED, Redis Lock, หรือ Reservation (available → reserved → sold แบบ atomic) How do you prevent the race condition? - SELECT ... FOR UPDATE SKIP LOCKED, a Redis lock, or a Reservation (available → reserved → sold, atomically).

คำตอบAnswer SELECT ... FOR UPDATE SKIP LOCKED ในหนึ่ง transaction ล็อก ticket ทันทีตอนค้นเจอ กันสองคนได้ใบเดียวกัน แล้วผูกกับ reservation lifecycle ด้านล่างเพื่อไม่ให้ ticket หายจาก pool ถ้าค้นแล้วไม่ซื้อ SELECT ... FOR UPDATE SKIP LOCKED in one transaction locks the ticket the instant it's found, preventing two users from getting the same one. Performance

วิเคราะห์ trade-off — search complexity, index, memory (เช่น Trie เร็วมากแต่ใช้ RAM เยอะ) Analyze the trade-offs - search complexity, index, memory (e.g. a Trie is very fast but uses a lot of RAM).

คำตอบAnswer ยิ่งรู้ตัวเลขในหลักไหนแน่นอนมากเท่าไหร่ ก็ยิ่งค้นเจอเร็วขึ้น เพราะ Postgres ใช้ index ของหลักนั้นตัดตัวเลือกทิ้งได้ทันที ส่วน pattern ที่เป็น * ล้วน (ไม่รู้เลยสักหลัก) ต้องไล่เช็คทุกแถว แต่ก็ยังเร็วพอเพราะข้อมูลมีแค่ 1 ล้านแถว The more digits you know for certain, the faster the search - Postgres uses that digit's index to rule out candidates instantly. A pattern that's all * (no digit known at all) has to check every row, but that's still fast enough since there's only 1M rows.

เลือกใช้ B-tree index เพราะเร็วพอสำหรับทุกกรณีข้างบน และกินความจำน้อย ไม่ต้องเก็บโครงสร้างข้อมูลพิเศษไว้ใน RAM ตลอดเวลา Chose B-tree indexes because they're fast enough for every case above and use little memory - no need to keep a special structure resident in RAM at all times.

02

Future Improvements

แนวทางขยับต่อถ้าระบบโตขึ้นหรือมี traffic จริงมาวัดผล — ยังไม่ต้องทำตอนนี้ Directions to move toward if the system grows or real traffic data comes in - not needed right now.

  • แยกเครื่อง Server รับเฉพาะการค้นหา — เพิ่มเครื่อง database สำรองที่คอยก็อปปี้ข้อมูลจากเครื่องหลักตลอดเวลา แล้วให้การค้นหาที่ยังไม่ถึงขั้นจอง ไปถามเครื่องสำรองนี้แทน ส่วนเครื่องหลักรับหน้าที่แค่ตอนต้อง "จองจริง" เท่านั้น เพราะในระบบจริงจำนวนคนค้นหาจะเยอะกว่าคนที่กดจองมาก การแยกแบบนี้ช่วยให้รองรับคนค้นหาพร้อมกันได้มากขึ้น โดยไม่ไปแย่งทรัพยากรกับเครื่องหลักที่ต้องดูแลเรื่องจองและ lock ส่วนข้อมูลที่เปลี่ยน (เช่น ticket ถูกจองไปแล้ว) จะไหลกลับจากเครื่องหลักไปอัปเดตเครื่องสำรองให้อัตโนมัติผ่าน replication ของ PostgreSQL เอง ไม่ต้องเขียนโค้ด sync เอง แต่จะมี delay สั้นๆ ระดับมิลลิวินาที Split off a read-only search server - add a standby database that continuously copies data from the primary, and route searches that haven't reached the reservation step to it instead. The primary only handles the actual "reserve" step, since real traffic has far more searches than reservations. This lets the system handle more concurrent searches without competing with the primary, which is busy with reservations and locks. Changed data (e.g. a ticket just reserved) flows back to the standby automatically via PostgreSQL's own replication - no sync code to write - though there's a short, millisecond-scale delay.
  • แบ่งข้อมูลเป็นก้อนตามเลขหลักแรก — ถ้าจำนวน ticket โตขึ้นมากๆ (เกิน 10 ล้านใบ) แบ่งข้อมูลออกเป็น 10 ก้อนตามเลขหลักแรกของ ticket ช่วยให้แต่ละก้อนมีขนาดเล็กลงและกระจายโหลดได้ pattern ที่ไม่ได้ระบุเลขหลักแรก เช่น ****23 ต้องกระจายไปถามทุกก้อนแล้วรวมผลลัพธ์เอง ซึ่งช้ากว่าการค้นในก้อนเดียว เหมาะทำตอนข้อมูลโตจริงจนเครื่องเดียวรับไม่ไหวเท่านั้น Shard data by leading digit - if the ticket count grows a lot (past ~10M), split the data into 10 shards by the ticket's first digit, keeping each shard smaller and spreading the load. A pattern that doesn't fix the leading digit, like ****23, has to fan out to every shard and merge the results, which is slower than a single-shard query - worth doing only once one machine genuinely can't keep up.
  • เก็บสถิติก่อนตัดสินใจเพิ่ม cache — ดูว่าคนค้นหาแบบไม่รู้เลขเลยสักหลักเกิดขึ้นบ่อยแค่ไหนจริงๆ ถ้าน้อยตามคาดไม่ต้องทำอะไรเพิ่ม ถ้าบ่อยค่อยสร้างระบบ cache ช่วยตามที่พูดถึงใน section performance Measure before adding a cache - track how often fully-wildcard searches actually happen in production. If rare as expected, no action needed; if common, build the pre-warming cache mentioned in the Performance section instead.
