# Functional Spec - User Management API

**Project:** 7solutions Full-Stack Assignment  
**Stack:** Go (Gin) · MongoDB · JWT (HS256)  
**Architecture:** Hexagonal (Ports & Adapters)

---

## 1. Overview

RESTful API for user management. Supports registration, login with JWT, and full CRUD operations on users. Uses MongoDB for persistence. Includes logging middleware and a background goroutine that logs total user count every 10 seconds.

---

## 2. User Model

```go
// internal/domain/user.go
type User struct {
    ID        primitive.ObjectID `bson:"_id,omitempty"    json:"id"`
    Name      string             `bson:"name"             json:"name"`
    Email     string             `bson:"email"            json:"email"`
    Password  string             `bson:"password"         json:"-"`       // never expose
    CreatedAt time.Time          `bson:"created_at"       json:"created_at"`
}
```

**Rules:**
- `Email` must be unique - enforce at application layer + unique index in MongoDB
- `Password` stored as bcrypt hash, never returned in response
- `ID` auto-generated by MongoDB (ObjectID)
- `CreatedAt` set at create time, never updated

---

## 3. Authentication Rules

| Rule | Detail |
|------|--------|
| Registration | Hash password before save. Return 409 if email already exists. |
| Login | Compare bcrypt hash. Return JWT on success. |
| JWT algorithm | HS256 with `JWT_SECRET` env var |
| JWT claims | `sub` = user ID (hex string), `exp` = 24h from issue |
| Protected endpoints | All `/users/*` routes require `Authorization: Bearer <token>` |
| Middleware | Validate token -> extract `sub` -> set in Gin context |

---

## 4. Business Rules

1. Email must match standard format (validated before save)
2. Name and Email are required for registration
3. Password minimum 6 characters
4. Update allows changing Name and/or Email only (not Password via this endpoint)
5. Delete removes the document from MongoDB (hard delete - spec does not mention soft delete)
6. List all users returns all documents (no pagination required by spec)
7. Background goroutine runs independently - does not block API
8. Logging middleware runs on every request - never skipped

---

## 5. Background Goroutine Behavior

```
Every 10 seconds:
  -> count documents in users collection
  -> log: "total users: N"
  -> if ctx.Done() received -> stop gracefully
```

Started in `main.go` after server is wired. Stopped by context cancel on SIGINT/SIGTERM.

---

## 6. Middleware Stack (in order)

```
Request
  -> Logger middleware    (log method + path + latency)
  -> Auth middleware      (JWT validate - applied to /users/* only)
  -> Handler
Response
```

## 7. Error Response Format

```json
{
  "success": false,
  "error": {
    "code": "NOT_FOUND",
    "message": "user not found"
  }
}
```

| HTTP | code | When |
|------|------|------|
| 400 | BAD_REQUEST | missing/invalid fields |
| 401 | UNAUTHORIZED | missing or invalid token |
| 404 | NOT_FOUND | user ID not in DB |
| 409 | CONFLICT | email already exists |
| 422 | VALIDATION_ERROR | schema validation failed |
| 500 | INTERNAL_ERROR | unexpected server error |
